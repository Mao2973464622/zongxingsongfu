# Ⅰ. 设计演变：从5屏到1屏

---

## 1.1 初始方案："粽有安康·灵境"（5屏叙事）

> 最初的野心：做一个**有叙事线的沉浸式小站**。

### 规划的五屏结构

| 屏 | 名称 | 核心玩法 | 当时设想的"亮点" |
|---|---|---|---|
| ① | 🚤 龙舟竞渡 | 手指划桨 / 控制龙舟前进 | 破水而出 loading，微交互入场 |
| ② | 🍙 粽星捧月 | 6个拟人粽子 → 点击弹祝福 | 甜·咸·辣·冰 口味人格化 |
| ③ | 🌿 艾草飘香 | 长按艾草束 → 祈福签文 | "安康驿"仪式感 |
| ④ | 🧵 五色丝线 | 拖拽/点击编五彩绳 | 传统"系丝线驱五毒" |
| ⑤ | 📜 安康寄语 | 自定义祝福 → 生成海报 | 分享出口 |

### ✅ 当时看起来很对的地方

- **叙事线通顺**：到站→品粽→祈福→系绳→带走，像翻一卷轴
- 视觉风格统一：艾草青 + 糯米白 + 朱砂红，新中式而非大红大绿
- 技术克制：零框架、纯静态、内联 SVG

### ❌ 但纸上谈兵的隐患

> 💡 **关键转折问题**（来自你自己）：*"用户一点开网址的时候，会有什么反应？"*

我们做了一次**三种典型用户的代入模拟**👇

#### 🧑 普通用户（30~45岁，刷到分享链接）

打开 → 2s龙舟破水loading → "这广告？"差点划走

→ 看到「轻触屏幕 启程」→ 点了 → 划了两下没反应 → 困惑 → 退出


**根因**：强制交互 + 缺少进度反馈 + 文案偏文雅（"启程"不如"点这里"直白）

#### 🧑‍🎤 年轻用户（爱玩H5）

动画不错 → 想跳过 → 找不到跳过

→ 划桨终于触发转场 → 突然跳屏② → "啊？"


**根因**：没有用户掌控感（不能跳过 = 被绑架），转场太硬

#### 👴 长辈用户（60+，子女发来）

书法字「粽有安康」→ 眯眼看不清

→ 不知道"轻触屏幕"什么意思 → 不动了


**根因**：字体辨识度 & 操作引导口语化不足

---

## 1.2 反思结论（诚实版）

> 方案方向对，但**把"创意完整度"放在了"用户第一秒感受"前面**。

具体来说：

- 🚫 **首屏门槛太高**：强制划桨 = 跳出率风险
- 🚫 **5屏 = 太长**：节日H5平均停留 < 30s，用户不会游完你的"展览馆"
- 🚫 **功能≠乐趣**：五彩绳屏花了最多代码，回报感最低——用户不会记住一根虚拟绳子
- 🚫 **缺少"无目的快乐"**：打开就该笑一下，而不是"学习怎么操作"

> 📎 一句话定性：**我们在做加法，用户在用减法筛选——只留一个值得点的东西。**

---

## 1.3 最终决策：砍到单屏「粽星送福」

### 砍法

| 原方案 | 处理方式 | 原因 |
|---|---|---|
| 屏① 龙舟 | ❌ 整屏删掉 | 强制交互劝退；精髓（水上氛围）降级为背景粒子 |
| 屏② 6粽 | ✂️ 压成 → **1个主角粽子** | 保留"点粽子得祝福"的核心爽点，去掉选择负担 |
| 屏③ 艾草签文 | ✂️ 合并进 | 祝福语本身就是签文，不用单独一屏 |
| 屏④ 五彩绳 | ❌ 直接砍 | 交互最重、记忆最浅 |
| 屏⑤ 海报 | ✂️ 换成 → **长按3秒弹祈福卡** | 截屏分享，零依赖，更快到达传播出口 |

### 新核心循环（10秒能讲清）

打开页面

→ 大粽子在飘 🍙（艾草叶背景粒子营造氛围）

→ 👆 点它 → 它抖 → 蹦出一句祝福（像拆盲盒）

→ 📱 摇一摇 → 粽子换表情

→ ⏳ 长按 → 弹出祈福卡 → 截屏分享


**为什么这个成立**：

- **零学习成本**：所有人知道"点一下"
- **即时反馈**：每次点击都有新东西（20条不重复）
- **可重复玩**：好奇心驱动"下句是什么"
- **传播路径短**：长按→截屏→发出去，三步完成

# Ⅱ. 最终成品功能详解

---

## 2.1 核心交互一览

| 操作 | 反馈 | 技术手法 |
|---|---|---|
| **点击粽子** | 弹性缩放弹跳 + 祝福语气泡弹出（2.5s消失） | `transform: scale()` + `cubic-bezier` 弹性 + `opacity` 气泡 |
| **摇一摇手机** | 粽子表情切换（😊→😂→🥰→😤循环） | `devicemotion` 加速度差值阈值 |
| **长按粽子 ≈2.6s** | 弹出居中祈福卡片（祝福语 + 署名 +「截屏提示」+ 关闭钮） | `setTimeout` 计时 + 动态 `div` 注入 |
| **背景氛围** | 12片 SVG 艾草叶 CSS `fall` 循环飘落 | 纯 CSS `@keyframes`，零 Canvas |

> 💡 **桌面端降级策略**：无陀螺仪时，每次点击粽子有 ~15% 概率随机换表情——不让桌面用户觉得"摇一摇功能坏了"。

# Ⅳ. 踩坑 & 决策记录（诚实复盘）

> 这部分才是**最有复用价值**的东西——不是"做了什么"，而是"为什么放弃什么"。

---

### 🔴 坑1：首屏强制交互 = 跳出邀请函

| | 原来 | 改法 |
|---|---|---|
| **现象** | 用户必须划桨N次才能进下一屏 | 砍掉整屏龙舟，粽子直接可见 |
| **根因** | 把"创意仪式感"优先级排在了"用户掌控感"前面 | 仪式感降级为背景氛围，不挡路 |
| **教训** | **节日H5的第1秒法则**：打开就要有东西可点，不能让用户在门口排队 | |

---

### 🔴 坑2：5屏很完整，但用户不会走完

| | 判断 |
|---|---|
| **幻觉** | "我做得多 = 显得用心" |
| **现实** | 多一屏 = 多一次流失机会。传播型小站，**出口越近越好** |
| **修正** | 把"品粽 + 祈福 + 分享"压进同一个闭环：点→笑→分享 |

---

### 🔴 坑3：长按保存卡 ≠ 一定要生成图片

| 方案A | 引入 `html2canvas` CDN → 能导出 PNG，但多一个网络依赖 + 偶发渲染差异 |
|---|---|
| **方案B（最终）** | 纯 DOM 卡片居中弹出 + 引导「截屏保存」→ **零依赖、100%稳定** |
| **结论** | 对一个 <30KB 的轻站，**去掉一个依赖 > 多一个功能** |

---

### 🔴 坑4：书法字体好看，但不能让它拖慢第一印象

| 做法 | 说明 |
|---|---|
| 仅 `h1` 用 `Ma Shan Zheng` | 标题一笔足矣，正文用 `ZCOOL KuaiLe`（更可读） |
| `font-display: swap` | 保证字体加载期间先用系统字，不白屏不卡 |

# Ⅶ. 经验总结（写给下一个节日小站）

> ### 🧠 最重要的一句话
> **节日 H5 卖的不是功能清单，是"打开那一刻的情绪"。**  
> 一个简单但让人愿意多点两下的东西，胜过一个精致但用户不敢碰的流程。

### 三条铁律（从这次迭代提炼）

1. **第1秒法则**：打开就要有东西可点，不能让用户在门口排队  
2. **减法 = 设计**：砍掉的屏不是浪费，是把注意力集中到真正有效的 10%  
3. **零依赖是特性，不是限制**：纯原生单文件 = 任何人、任何设备、任何时间都能打开
